# AGENTS.md — 이 저장소에서 작업하는 모든 에이전트/협업자용 규칙

작성일: 2026-03-22

---

## 1. 최상위 원칙

이 저장소의 모든 작업은 반드시 아래 두 문서를 기준으로 한다.

1. `PRD.md`
2. `IMPLEMENTATION.md`

기능을 추가하거나 수정할 때는 먼저 두 문서를 읽고,
현재 작업이 문서와 일치하는지 확인한다.

---

## 2. 문서 동기화 규칙

아래 경우에는 작업과 함께 문서를 반드시 업데이트한다.

- 제품 목표가 바뀐 경우
- 문제 은행이 바뀐 경우
- 데이터 스키마가 바뀐 경우
- API 계약이 바뀐 경우
- 배포 구조가 바뀐 경우
- 보안 정책이 바뀐 경우
- UI 흐름이 바뀐 경우

### 필수 지시사항
- **항상 `PRD.md` 와 `IMPLEMENTATION.md` 를 기준으로 작업한다.**
- **작업이 끝날 때마다 GitHub 에 push 한다.**
- **작업이 끝날 때마다 `PRD.md` 와 `IMPLEMENTATION.md` 를 최신 상태로 업데이트한다.**

이 세 문장은 예외 없이 지켜야 한다.

---

## 3. 작업 시작 전 체크리스트

1. `PRD.md` 확인
2. `IMPLEMENTATION.md` 확인
3. 현재 작업 범위를 한 줄로 정리
4. 영향 받는 파일 목록 예상
5. 문서 업데이트 필요 여부 먼저 판단

---

## 4. 절대 어기면 안 되는 구현 규칙

### 4-1. 수학 계산
- 부동소수점 기반 임시 계산을 금지한다.
- 모든 유리수 계산은 정확한 분수 연산으로 처리한다.
- 부호 해석은 문자열 땜질이 아니라 명시적 파서/정규화 로직으로 처리한다.

### 4-2. 보안
- Google 인증 정보는 서버 전용으로만 둔다.
- 서비스 계정 키를 클라이언트 번들에 포함시키지 않는다.
- 비밀값을 저장소에 커밋하지 않는다.

### 4-3. 로깅
- 시도 기록은 append-only 로 남긴다.
- 기존 학생 시도 기록을 덮어쓰지 않는다.
- 단계별 도전 회차를 보존한다.

### 4-4. 콘텐츠
- 문제 은행을 임의로 바꾸지 않는다.
- 문제 수정 시 `QUESTION_BANK.md` 와 `problem-bank.json` 을 함께 고친다.
- 난이도 기준을 PRD 와 다르게 바꾸지 않는다.

### 4-5. 사용자 경험
- 실패 피드백은 비난형 문구를 쓰지 않는다.
- 색만으로 정보 전달하지 않는다.
- 모바일/태블릿 사용성을 항상 확인한다.

## 5. 브랜치 / 커밋 / push 규칙

### 브랜치 이름 예시
- `feat/rabbit-parser-step-validation`
- `feat/google-sheets-append-logging`
- `fix/number-line-range-bug`
- `docs/prd-implementation-sync`

### 커밋 메시지 예시
- `feat: add rabbit parser split-step validator`
- `fix: correct rational number line padding`
- `docs: sync PRD and implementation with logging changes`

### push 규칙
- 의미 있는 작업 단위가 끝나면 바로 push
- 큰 기능은 중간 단계라도 동작하는 상태에서 push
- 문서 업데이트 없는 기능 push 금지

## 6. 완료 정의

작업 완료로 간주하려면 아래를 모두 만족해야 한다.

- 코드가 로컬에서 동작한다
- 관련 테스트가 통과한다
- 영향 받는 문서가 업데이트되었다
- GitHub 에 push 되었다
- 후속 작업자가 이해할 수 있을 만큼 변경 이유가 남아 있다

## 8. 리뷰 기준

### 수학적 정확성
- 정답만 맞는지 보지 말고 과정 검증이 일관적인지 확인
- `-(-a)`, `-(+a)`, 분수/소수 혼합 케이스 우선 확인

### 제품 일관성
- PRD 흐름과 화면이 맞는지 확인
- 난이도 정의가 지켜졌는지 확인

### 기술 일관성
- 타입 안전성 유지
- 서버/클라이언트 책임 분리
- Google Sheets 호출이 과도하지 않은지 확인

## 10. 문서 우선 충돌 해결 규칙

코드와 문서가 충돌하면 다음 순서로 판단한다.

1. PRD 의 학습 목표
2. IMPLEMENTATION 의 구현 원칙
3. QUESTION_BANK 의 콘텐츠 정의
4. 현재 코드

코드가 문서와 다르면,
먼저 코드가 틀렸는지 확인하고,
제품 결정이 바뀐 것이 맞다면 문서를 먼저 고친 뒤 코드를 수정한다.
