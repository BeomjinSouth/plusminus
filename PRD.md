# PRD — 유리수의 덧셈과 뺄셈 3모형 게이미피케이션 웹앱

작성일: 2026-03-22  
문서 상태: 구현 반영 v3

---

## 1. 제품 한 줄 정의

학생이 **셈돌 모델**, **카드 점수 미션(우체부 모델)**, **사용자 제작 모델(토끼/부호-분해 모델)** 을 오가며  
정수와 유리수의 덧셈·뺄셈을 **직관 → 이야기 → 기호 조작** 순서로 익히도록 돕는  
중학교용 웹앱을 만든다.

---

## 2. 왜 이 제품이 필요한가

정수와 유리수의 덧셈·뺄셈은 학생들이 다음 이유로 자주 막힌다.

- 연산 기호의 `+`, `-` 와 수의 부호 `+`, `-` 를 서로 혼동한다.
- `a - (-b)` 와 같은 표현을 절차로만 외우고, 왜 그런지 이해하지 못한다.
- 수직선, 조작 활동, 기호식 계산이 서로 연결되지 않아 유형이 바뀌면 다시 틀린다.
- 분수나 소수까지 들어가면 부호 판단과 계산이 동시에 어려워진다.

이 앱은 같은 수학 개념을 서로 다른 표현 모델로 반복 경험하게 하여,
학생이 한 가지 풀이 요령이 아니라 **구조 자체**를 익히게 하는 것을 목표로 한다.

## 3. 목표 사용자

### 3-1. 1차 사용자
- 중학교 1학년 학생
- 정수와 유리수의 덧셈·뺄셈을 처음 배우는 학생
- 부호 처리와 괄호 정리에 약한 학생
- 게임처럼 짧은 성공 경험을 자주 필요로 하는 학생

### 3-2. 2차 사용자
- 수학 교사
- 학급 단위로 문제 풀이 기록을 남기고 싶은 교사
- 문항별 재도전 기록, 정오답, 단계별 실패 지점을 보고 싶은 교사

## 4. 학습 설계의 큰 원칙

1. **같은 개념을 다른 표현으로 보게 한다.**  
   돌, 이야기, 수직선/기호 조작을 오가게 한다.

2. **한 번에 한 가지 인지 부담만 준다.**  
   예를 들어 사용자 제작 모델에서는  
   먼저 끊기, 다음에 부호 정리, 마지막에 이동과 합산으로 나눈다.

3. **정답만 맞히는 것이 아니라 과정도 평가한다.**  
   어느 단계에서 틀렸는지가 Google Sheets 에 남아야 한다.

4. **유리수는 정확한 값으로 다룬다.**  
   부동소수점 오차를 피하기 위해 분수 기반의 정확한 연산을 사용한다.

5. **문항은 고정형(problem bank)으로 운영한다.**  
   생성형이 아니라 이미 정해진 문항 세트를 푸는 구조로 간다.

## 5. 세 가지 모형에 대한 연구 정리와 제품 해석

> 중요한 전제  
> 문헌에서 셈돌 모델과 우체부 모델은 대체로 **정수 지도** 맥락에서 많이 다뤄진다.  
> 따라서 이 PRD 에서 제시하는 **유리수 확장 방식**은 문헌의 정식 표준이라기보다  
> 본 앱을 위한 **제품 설계 제안**이다.

### 5-1. 셈돌 모델

#### 의미
- 두 가지 색의 돌로 양수와 음수를 나타낸다.
- 같은 수의 양돌과 음돌이 만나면 `0` 으로 소거되는 구조를 드러낸다.
- 덧셈은 합치기, 뺄셈은 제거하기로 이해하게 돕는다.

#### 사용하는 경우
- `(+a)+(-b)` 같이 서로 다른 부호가 만나 소거되는 상황
- `a-(-b)` 를 설명하기 위해 **없는 쌍(0쌍)** 을 추가해야 하는 상황
- 부호가 다를 때 절댓값의 차가 왜 생기는지 시각적으로 보여주고 싶을 때

#### 사용하는 의도와 유용성
- `반대수`와 `0의 쌍`을 매우 직관적으로 보여준다.
- 손으로 끌고 놓는 상호작용과 잘 어울린다.
- 정답을 외우기보다 “왜 없어졌는지”를 보게 만든다.
- 학생이 “빼는 수를 더한다”를 기계적으로 외우기 전에 조작 경험을 갖게 한다.

#### 한계
- 뺄셈에서 `없는 쌍을 만들어 넣는다`는 발상이 처음에는 오히려 어렵게 느껴질 수 있다.
- 연산 기호의 `-` 와 음수 부호의 `-` 가 섞이면 학생이 혼동할 수 있다.
- 곱셈, 나눗셈, 복잡한 유리수 전체를 그대로 옮겨가기에는 덜 자연스럽다.
- 분수 문제에서는 돌 하나의 값이 `1/n` 으로 바뀌는 설계가 필요하므로,
  실물 감각이 약해질 수 있다.

#### 이 앱에서의 해석
- 정수 문제에서는 돌 1개 = 1
- 분수 문제에서는 돌 1개 = `1/공통분모`
- 예: `(+2/3)+(-1/3)` 은 `+돌 2개`와 `-돌 1개`의 조작으로 표현

### 5-2. 우체부 모델(학생 노출명: 카드 점수 미션)

#### 의미
- 우체부가 **가져오는 것**을 덧셈, **가져가는 것**을 뺄셈으로 해석한다.
- 어음은 양수, 고지서는 음수로 보는 식으로  
  **연산 기호**와 **수의 부호**를 이야기 속 역할로 분리해 준다.

#### 사용하는 경우
- `+(-5)`, `-(-5)` 같은 표현을 이야기로 번역하고 싶을 때
- “음수를 빼는 것”이 실제로는 어떤 변화를 뜻하는지 설명하고 싶을 때
- 학생이 부호를 상징 조작이 아니라 사건으로 이해하게 하고 싶을 때

#### 사용하는 의도와 유용성
- `빼기`를 단순 계산 부호가 아니라 “가져가기”라는 행동으로 보게 해 준다.
- `-(-5)` 를 “벌점을 없애 간다”처럼 해석하면 양의 변화가 자연스러워진다.
- 부호의 중첩을 서사적으로 설명하는 데 강하다.

#### 한계
- 어음, 고지서 같은 어휘가 학생에게 낯설 수 있다.
- 이야기 이해가 수학 이해보다 더 큰 부담이 되면 본말이 바뀔 수 있다.
- 나눗셈까지 자연스럽게 확장하기는 어렵다.
- 유리수로 확장할 때 `3/4원` 같은 맥락은 어색할 수 있으므로,
  실제 제품에서는 **포인트 카드 / 보상 카드 / 벌점 카드** 로 바꾸는 편이 낫다.

#### 이 앱에서의 해석
- 원래 모델의 구조는 유지하되, 학생 친화적으로  
  `보상 카드(+), 벌점 카드(-)` 또는 `코인 카드(+) / 패널티 카드(-)` 로 시각화한다.
- 학생에게는 `우체부` 라는 용어를 거의 노출하지 않고, 기본 명칭을 **카드 점수 미션** 으로 둔다.
- 카드가 점수판 안팎으로 움직이는 애니메이션과 장면 스티커 로그를 통해 식을 사건으로 번역한다.

### 5-3. 사용자 제작 모델(토끼/부호-분해 모델)

#### 의미
- 식을 **항 단위로 끊고**
- 각 항 안의 `-` 개수를 보고 최종 부호를 결정한 뒤
- 정리된 항을 수직선에서 토끼가 이동하며 계산하는 방식이다.

#### 사용하는 경우
- `-(+3)-(-5)+7` 같이 괄호와 부호가 여러 번 겹치는 식
- 학생이 부호 정리 규칙을 기호 수준에서 명확히 익혀야 할 때
- 조작 모델에서 상징 조작으로 넘어가는 다리 역할이 필요할 때

#### 사용하는 의도와 유용성
- 식을 한꺼번에 보지 않고 **항 단위**로 구조화하도록 돕는다.
- `부호 결정`과 `크기 합산/차이`를 분리하여 생각하게 한다.
- 수직선 이동과 결합하면 계산 순서가 눈에 보인다.
- 학생이 어디서 틀렸는지 세밀하게 기록하기 좋다.
  - 끊기에서 틀림
  - 부호 정리에서 틀림
  - 이동 방향에서 틀림
  - 최종 합에서 틀림

#### 한계
- 너무 자주 사용하면 절차만 따라하는 규칙 암기로 흐를 수 있다.
- 의미 이해 없이 `- 개수 세기` 만 남을 위험이 있다.
- 복잡한 분수 계산에서는 부호는 맞지만 산술 계산이 추가로 어려워질 수 있다.

#### 이 앱에서의 해석
- 이 모델은 앱의 **상징 조작 핵심 모드**로 둔다.
- 셈돌/우체부를 먼저 경험한 뒤,  
  같은 문제를 토끼 모델로 옮겨오게 하는 연결 설계를 권장한다.

## 6. 제품 목표

### 6-1. 학습 목표
- 학생이 양수/음수의 의미를 구분한다.
- 학생이 덧셈과 뺄셈에서 연산 기호와 수의 부호를 구분한다.
- 학생이 `a-b = a+(-b)` 를 맥락과 절차 양쪽에서 이해한다.
- 학생이 정수뿐 아니라 간단한 유리수(분수, 소수)로 확장해도 부호 처리를 유지한다.
- 학생이 중간 과정을 설명 가능한 상태에 도달한다.

### 6-2. 제품 목표
- 짧은 몰입 구간으로 반복 학습이 가능하다.
- 학생 식별 정보와 문항별 시도 기록이 Google Sheets 에 남는다.
- 교사가 별도 DB 없이도 학급 수준의 기록을 볼 수 있다.
- 브라우저만 있으면 수업 시간에 바로 사용할 수 있다.

## 8. 핵심 사용자 흐름

### 8-1. 입장
1. 학생이 학교, 학년, 반, 번호를 입력
2. 세션 생성
3. 모델 선택 화면으로 이동

입장 화면은 수업 중 빠르게 접근할 수 있도록
- 별도 홍보성 랜딩 없이 단일 로그인 카드로 바로 시작하고
- 제목은 `정수와 유리수의 덧셈 뺄셈 연습` 으로 고정하며
- 학교, 학년, 반, 번호 입력과 단일 주요 CTA 만 남겨
빠른 입력 흐름에 집중한다.

첫 화면은 중앙 정렬된 로그인 카드 1개만 두고,
모델 소개와 상세 선택은 세션 시작 후 로비에서 처리한다.

### 8-2. 모델 선택
- 셈돌 모델
- 카드 점수 미션(내부 모델 ID: `postman`)
- 토끼/부호-분해 모델

### 8-3. 세트 선택
각 모델마다 3세트
- 하
- 중
- 상

각 세트는 5~10문항으로 구성하고, 초안에서는 **7문항**으로 설계한다.

### 8-4. 문제 풀이
- 문제 시작
- 단계별 상호작용 수행
- 즉시 피드백
- 정답이면 다음 단계 또는 다음 문항
- 오답이면 같은 단계 재도전
- 각 도전 회차와 정오답이 기록됨

현재 구현에서는 `play/[model]/[difficulty]` 화면에서 한 세트의 7문항을 순차 진행한다.

### 8-5. 세트 종료
- 점수, 정확도, 연속 정답, 재도전 횟수 표시
- 같은 세트 재도전 또는 다른 모델 이동

## 9. 모델별 상세 경험 설계

## 9-1. 셈돌 모델

### 기본 인터랙션
- 왼쪽 패널: 식
- 가운데 보드: 조작 공간 + 양돌/음돌이 실제로 쌓이는 시각 보드
- 아래 트레이: `+돌`, `-돌`, `0쌍 추가` 버튼 + 현재 항에 놓을 돌 미리보기
- 우측 패널: 현재 남은 돌 수, 예상 결과 입력칸

### 단계
1. 항별로 양돌/음돌 놓기
2. 사라질 `0쌍` 개수 예측
3. 남은 돌이 뜻하는 결과 제출

MVP 구현에서는 `0쌍 추가`를 직접 조작하기보다,
항별 배치와 소거 쌍 예측을 통해 구조를 먼저 익히게 한다.

### 피드백
- 잘 놓은 경우: 준비한 돌이 보드에 부드럽게 나타남
- 잘못 놓은 경우: 해당 조작만 되돌리고 이유를 한 줄로 제시
- 되돌리기: 마지막에 놓은 돌만 짧게 사라짐
- 소거 성공: 짝이 맞는 돌이 반짝이며 사라짐

### 게임화
- `0쌍 마스터`
- `한 번에 소거 성공`
- `완벽 클리어`

## 9-2. 우체부 모델(학생 노출명: 카드 점수 미션)

### 기본 인터랙션
- 상단에 식과 진행 상황을 크게 제시
- 점수판 무대에서 카드가 들어오거나 나가는 애니메이션 노출
- `보상 카드(+)`, `벌점 카드(-)` 장면 카드 4종 선택
- 왼쪽에 맞힌 장면이 스티커처럼 쌓이는 장면 모음 패널 제공

### 단계
1. 현재 식을 이야기로 읽기
2. 네 가지 행동 카드 중 맞는 장면 선택
3. 누적 변화 기록 확인
4. 최종 값을 제출

### 피드백
- 행동과 부호가 맞으면 카드가 무대에서 움직이며 장면 모음에 추가
- 틀리면 앞 기호와 카드 종류를 분리해 다시 읽게 하는 짧은 질문형 피드백 제시

### 게임화
- `장면 번역 성공`
- `부호 번역 성공`
- `실수 없는 카드 마스터`

## 9-3. 사용자 제작 모델(토끼/부호-분해 모델)

### 단계 구조
이 모델은 현재 구현에서 5단계로 고정한다.

#### 1단계: 끊기
예: `-(+3)-(-5)+7`

학생은 아래처럼 끊는 위치를 클릭해야 한다.

- `-(+3)`
- `-(-5)`
- `+7`

정답이면 다음 단계로 이동한다.

#### 2단계: 항별 최종 부호 정리
- `-(+3)` → `-3`
- `-(-5)` → `+5`
- `+7` → `+7` 또는 `7`

학생은 각 항을 직접 입력한다.

#### 3단계: 토끼 이동 계산
- 학생이 시작 위치를 클릭
- 각 항을 순서대로 적용
- 음수면 왼쪽, 양수면 오른쪽
- 엔터 또는 확인 버튼으로 이동 완료 제출

예:
- 시작 0
- `-3` 적용 → 0에서 왼쪽 3칸
- `+5` 적용 → 현재 위치에서 오른쪽 5칸
- `+7` 적용 → 현재 위치에서 오른쪽 7칸

#### 4단계: 최종 값 제출
- 토끼가 마지막에 멈춘 위치를 최종 값으로 다시 적는다.

### 교육적 장점
- 구문 분석, 부호 해석, 합산을 분리해 학습시킨다.
- 단계별 로그가 매우 잘 남는다.
- 수직선과 기호식의 연결이 강하다.

### 설계 유의점
- 규칙 설명이 지나치게 기계적이지 않도록
  각 단계마다 짧은 이유 설명 툴팁을 둔다.
- “왜 오른쪽으로 갔는가?” 를 확인하는 반성 질문을 세트 마지막에 넣는다.

## 10. 문제 구성 원칙

### 10-1. 전체 구조
- 모델 3종
- 각 모델 3세트
  - 하
  - 중
  - 상
- 세트당 7문항 초안

총 63개 플레이 슬롯  
(수학적으로는 일부 문항을 모델별로 재사용할 수 있음)

### 10-2. 난이도 기준
#### 하
- 정수 중심
- 항 2~3개
- 괄호 거의 없음
- 분수는 없음 또는 소수 1개 정도

#### 중
- 항 3개
- 정수 + 간단한 분수/소수
- 같은 분모 또는 작은 공통분모
- 부호 혼합

#### 상
- 괄호 포함
- `-(+a)`, `-(-b)` 포함
- 분수/소수 혼합 가능
- 토끼 모델의 끊기 단계가 의미 있게 작동해야 함

### 10-3. 유리수 확장 규칙
- 공통분모가 12를 넘지 않게 제한
- 소수는 유한소수 위주
- 한 문제 안에서 너무 많은 표현 혼합을 피함
- 수직선 눈금은 문제별 최소공배수에 따라 자동 결정

## 15. 주요 위험과 대응

### 위험 1. Google Sheets 쓰기 횟수 과다
**대응:** 단계별 로그를 바로 전송하되, 한 요청에 여러 이벤트를 묶어 append

### 위험 2. 학생이 부호 규칙만 암기
**대응:** 셈돌 → 우체부 → 토끼 순서의 추천 학습 루프 제공

### 위험 3. 유리수 수직선 눈금이 너무 촘촘해짐
**대응:** 문제 은행 단계에서 공통분모 제한, 자동 줌, 단계별 하이라이트 적용

### 위험 4. 우체부 이야기 어휘가 낯섦
**대응:** 실제 UI 에서는 `우체부` 용어를 숨기고, 보상/벌점 카드가 점수판에 드나드는 `카드 점수 미션` 으로 치환

### 위험 5. 서비스 계정 키 노출
**대응:** 서버 전용 환경 변수 사용, 키를 클라이언트에 절대 노출하지 않음
